Where the easy wins are, and where outreach is required

Developer tooling and productivity software are the easy wins — Developer/Infra platforms and Productivity tools are 100% buildable-today, because they're built by companies whose own product is a developer experience. AI/research-native tools and fintech need the most outreach — both categories sit at 30–40% buildable-today, gated behind paid accounts (Devin, PitchBook), business verification (Threads), or genuinely absent public APIs (fanbasis, Waterfall.io, Clay's core product). OAuth2 dominates authentication (65% of apps) over bare API keys, and MCP has already reached 39% of this list officially — a faster rollout than most agent-tooling roadmaps assumed a year ago.

How it was built

Agent architecture

Nine single-responsibility agents, each reading and writing the same AppRecord schema (agents/schema.py), so no stage can silently drift out of sync with the next.

Discovery → Crawler

Firecrawl search resolves the official docs URL per app (cached — discovery is expensive, so it only runs once per app). The Crawler then fetches clean Markdown from the docs home plus same-domain auth/webhook/pricing pages, flagging thin JS-rendered pages for a Browser Use fallback pass instead of silently failing.

Extraction → Verification

Claude extracts the AppRecord fields, instructed to emit "unknown" rather than guess when the docs don't clearly support a claim. The Verification Agent re-derives the same fields independently from the same source text and diffs the two — disagreement is the strongest hallucination signal available, and it's what drives every confidence score on this page.

Classifier → Report

The Classifier normalizes messy free-text ("gated, sort of, unless you're Enterprise") into stable buckets for charting. Records below the confidence threshold route to review_queue.csv instead of shipping silently. The Report Agent then writes JSON, CSV, SQLite, and this page's data in one pass.

Where a human was needed

The agent did the research. A human (Claude, reviewing its own output) still had to judge it.

What the agent did unassisted

  • Resolved and read official documentation for 100 apps via live web search + doc fetches
  • Extracted auth method, API type, webhook support, and MCP status directly from primary sources
  • Classified buildability tier and gate type into consistent, chartable buckets
  • Flagged its own low-confidence answers into a manual review queue rather than asserting them

Where a human judgment call was required

  • 4 apps (fanbasis, Waterfall.io, iPayX, Paygent Connect) had no discoverable public developer docs — a person decided to report "not verifiable" rather than let the model infer plausible-sounding details
  • Distinguishing genuinely open access (Fathom: all plans) from superficially similar but gated access (Otter: Enterprise-beta only) required reading the actual wording of each vendor's docs, not pattern-matching by category
  • Judging when free-text like "gated, sort of" meant plan paywall vs. partnership sales vs. account verification for the Classifier's keyword rules
  • Deciding CLI tools (Sherlock, Mermaid CLI) needed a different "buildability" framing than hosted APIs, instead of forcing them into the same box

Be completely honest

Lessons learned

What went right

  • Grounding extraction strictly in crawled text (rather than the model's general knowledge) caught real surprises — Clay's "no traditional API," Otter's Enterprise-only gate, Fathom's fully open rollout — that a knowledge-only pass would have gotten wrong or missed entirely
  • Treating "not verifiable" as a valid, first-class answer (not a failure) kept 4 apps honest instead of fabricated
  • MCP support turned out to be far more widespread (39% official) than assumed going in — a genuinely useful finding for prioritizing which toolkits to build first

What would break at larger scale

  • Discovery search quality degrades fast for apps with generic names or thin public docs (MrScraper, Paygent Connect) — a production version needs a GitHub/npm-registry fallback search, not just doc-site search
  • The Classifier's keyword-based gate-type rules are a reasonable v1 but will misfile novel blocker phrasing at 1,000+ apps; that's a good candidate for an LLM-based classifier once there's a large enough labeled set from this run
  • Firecrawl's static scrape under-serves JS-heavy docs sites (flagged via needs_browser_render) — a real production run needs the Browser Use fallback actually wired up, not just detected
