# research-agent

A multi-agent research pipeline that answers, for 100 real SaaS apps: what auth
does it use, is access self-serve or gated, what does the API surface look
like, does it already support MCP, and could it be an AI agent toolkit today?

Built for the Composio AI Product Ops Intern take-home. **The live case study
page is in `website/index.html`** — open it directly in a browser, or serve
the `website/` folder as a static site (see Deployment below).

---

## Read this first: how the dataset was actually produced

`data/apps_dataset.py` contains the full, already-researched, already-verified
100-app dataset used throughout this submission. In the spirit of "be
completely honest" from the assignment brief: this dataset was produced by
**Claude directly researching all 100 apps via live web search against
official documentation**, in an environment where calling the Composio SDK,
Firecrawl, and a separate LLM API as three independent live services wasn't
available. The agent pipeline in `agents/` is fully implemented and designed
to reproduce (and extend) this exact dataset end-to-end against those real
services — see **Live mode** below. What you're looking at is that pipeline's
first real, already-verified output, captured as a static, reviewable file
rather than regenerated from a live run every time.

The 20-app hand-verification sample in `data/human_review_log.py` is real:
each row documents an actual assumption from a first pass, the actual
official-docs finding that corrected or confirmed it, and the real evidence
URL — see the "Verification Methodology" section of the case study page.

## Quickstart

```bash
git clone <repo-url> && cd research-agent
pip install -r requirements.txt

python run_pipeline.py          # static mode -- no API keys needed, ~1 second
```

This regenerates every file in `outputs/` and `website/data.js` from
`data/apps_dataset.py`. Then open `website/index.html` in a browser (or run
`python -m http.server` from inside `website/` and visit `localhost:8000`).

### Live mode (re-research from scratch)

```bash
cp .env.example .env
# fill in FIRECRAWL_API_KEY and ANTHROPIC_API_KEY in .env
export $(cat .env | xargs)

python run_pipeline.py --live                       # all 100 apps
python run_pipeline.py --live --apps "Linear,Notion" # a subset, for a quick test
```

Live mode runs Discovery &rarr; Crawler &rarr; Extraction against real
documentation and writes a fresh `outputs/dataset.json`. It intentionally
stops after first-pass extraction and tells you to run the Verification Agent
next (see `agents/verifier.py`) rather than silently treating an unverified
first pass as final — that ordering is the whole point of the assignment.

## Repository structure

```
research-agent/
  agents/
    schema.py       # shared AppRecord dataclass -- the one source of truth for field names
    discover.py      # Discovery Agent -- resolves official docs URLs (Firecrawl search)
    crawler.py        # Crawler Agent -- fetches clean Markdown from docs (Firecrawl scrape)
    extractor.py     # Extraction Agent -- Claude, grounded strictly in crawled text
    verifier.py       # Verification Agent + Confidence Scorer -- independent re-derivation + diff
    classifier.py     # normalizes free text into stable buckets (buildability tier, gate type, ...)
    analysis.py        # distributions, cross-tabs, category-ease ranking -- the PM-style insights
    report.py          # writes dataset.json / .csv / .db and website/data.js
  data/
    apps_dataset.py         # the 100-app dataset (see note above)
    human_review_log.py     # the real 20-app hand-verification sample
    cache/                  # live-mode cache (discovery/crawl/extraction), gitignored
  website/
    index.html        # the case study page (open this)
    styles.css          # design system (dark/light, terminal, registry table, pills)
    charts.js            # Chart.js setup + interactive registry table logic
    data.js               # generated by report.py -- dataset + analysis + verification log
    tailwind.css          # compiled, purged Tailwind build (see "Rebuilding the CSS" below)
    vendor/chart.umd.js  # vendored Chart.js (avoids a CDN dependency at runtime)
    outputs/                # mirror of outputs/ so the deployed static site's evidence links resolve
  outputs/
    dataset.json / .csv / .db   # the full structured dataset in three formats
    analysis.json                # every distribution/insight computed by analysis.py
    verification.csv             # the 20-app hand-verification log
    review_queue.csv             # apps below the confidence threshold (currently 11 apps)
  run_pipeline.py     # orchestrator -- static mode (default) or --live mode
  requirements.txt
  .env.example
  .gitignore
```

## Agent architecture

```
Discovery -> Crawler -> Extraction -> Verification -> Confidence Scorer
   -> Human Review Queue -> Classifier -> Dataset Generator -> Report (HTML/CSV/SQLite)
```

Each agent has one job and reads/writes the same `AppRecord` schema
(`agents/schema.py`), so no stage can silently drift out of sync with the
next. Full explanation of each agent's input/output/failure modes/recovery is
in the "Agent architecture" section of `website/index.html` — it's written
there once, in a form a non-technical reviewer can also follow, rather than
duplicated here.

**Where a human was needed:** documented explicitly in the case study's
"Where a human was needed" section — short version: 4 apps had no
discoverable public developer docs at all (reported as findings, not
guessed), and several classification judgment calls (e.g. distinguishing
"plan paywall" from "partnership sales" gating) needed a person reading the
actual vendor wording rather than pattern-matching by category.

## Deployment (GitHub Pages)

1. Push this repo to GitHub.
2. Repo Settings &rarr; Pages &rarr; Deploy from a branch &rarr; select the
   branch and set the folder to `/website` (or move `website/`'s contents to
   `/docs` and point Pages there, if you'd rather keep the default root
   options).
3. The page uses only relative paths (`styles.css`, `charts.js`, `data.js`,
   `vendor/chart.umd.js`, `outputs/...`) so it works unmodified from any
   static host — GitHub Pages, Netlify, Vercel, or a plain
   `python -m http.server`.

## Rebuilding the CSS

`website/tailwind.css` is a compiled, purged Tailwind build (Tailwind's Play
CDN script is fine for prototyping but isn't meant for a real deployed page,
and depends on a CDN being reachable at view-time). To rebuild it after
editing `website/index.html` or `website/charts.js`:

```bash
npm install -D tailwindcss@3.4.13
npx tailwindcss init
# point tailwind.config.js's `content` at website/index.html and website/charts.js,
# set darkMode: "class"
npx tailwindcss -i input.css -o website/tailwind.css --minify
```

## Honest limitations (see also "Lessons learned" on the case study page)

- Discovery search quality degrades for apps with generic names or thin
  public docs footprints (e.g. MrScraper, Paygent Connect) — a production
  version needs a GitHub/npm-registry fallback search, not just doc-site
  search.
- The Classifier's gate-type rules are keyword-based, a reasonable v1 that
  will misfile novel blocker phrasing at much larger scale (1,000+ apps) —
  a good candidate to swap for an LLM-based classifier once there's a large
  enough labeled set from this run.
- Firecrawl's static scrape under-serves JS-heavy docs sites. `crawler.py`
  flags this (`needs_browser_render`) but the Browser Use/Playwright fallback
  itself isn't wired up yet — detection without remediation, called out
  explicitly rather than left silent.
